1. ВВЕДЕНИЕ И ИСТОРИЯ CBLERRV5.1
═══════════════════════════════════════════════════════════════════════════════

CBlerr версия 5.1 – это революционный язык программирования, который был
разработан с целью предоставить разработчикам идеальный баланс между удобством
синтаксиса и производительностью. Название "CBlerr" происходит от сочетания "C"
(язык программирования с низким уровнем абстракции) и "Blurr" (размытие,
объединение). Этот язык буквально объединяет лучшие черты нескольких
популярных языков программирования.

История развития CBlerr началась несколько лет назад, когда разработчики
осознали, что существует огромная пропасть между легкостью написания кода на
высокоуровневых языках вроде Python и необходимостью получить максимальную
производительность, которую может дать только язык низкого уровня вроде C.

CBlerr была создана как ответ на этот вызов: "Почему бы не иметь язык, который
позволяет писать код с синтаксисом Python, но компилируется в машинный код
с производительностью C, и при этом позволяет писать встроенный ассемблер
для критичных участков?"

Версия 5.1 представляет собой зрелую реализацию этой концепции, с полной
поддержкой современных парадигм программирования, включая функциональное
программирование, универсальное программирование (generics), и модульную
архитектуру. После пяти версий развития, язык достиг уровня стабильности,
надёжности и производительности, позволяющей использовать его для создания
высококритичного кода, от операционных систем до встраиваемых систем и
трёхмерных игровых движков.

2.1 ПАРАДИГМА "PYTHON С ЗУБАМИ"

CBlerr следует парадигме, которую можно охарактеризовать как "Python с зубами".
Это означает, что язык предоставляет простоту и читаемость Python, но при этом
имеет "зубы" - то есть производительность и низкоуровневый контроль C.

Язык основан на следующих принципах:

• Явность (explicit is better than implicit) - всегда показывайте типы и намерения.
  Неявное поведение скрывает ошибки и усложняет отладку. В CBlerr все типы
  должны быть явно указаны.

• Полный контроль над деталями реализации - нет скрытых механизмов. Вы знаете
  как происходит выделение памяти, как работают вызовы функций, какие регистры
  используются для параметров.

• Никаких скрытых копий или перемещений данных - вы контролируете всё. Если
  вам нужна копия, вы сами её делаете. Если вам нужно переместить данные,
  вы это явно указываете.

• Предсказуемое использование памяти - вы знаете, что происходит. Нет случайных
  выделений памяти garbage collector-ом, которые могут заморозить вашу программу
  на критичный момент.

• Детерминированная производительность - нет сюрпризов с garbage collection или
  неожиданным замедлением. Если ваш код работает за 1 миллисекунду, он всегда
  будет работать за 1 миллисекунду (в пределах того же оборудования).

2.2 МОДЕЛЬ КОМПИЛЯЦИИ И ЦЕПЬ ПРЕОБРАЗОВАНИЙ

CBlerr использует многоэтапную цепь компиляции для превращения исходного
кода в высокооптимизированный машинный код. Каждый этап выполняет определённую
функцию и передаёт результат следующему этапу:

Этап 1 - Лексический анализ (Lexing):
    Исходный текст CBlerr файла разбивается на токены. Лексер считывает
    символы и распознаёт ключевые слова, идентификаторы, операторы, строки
    и числа. Результат: последовательность токенов, готовая для парсера.

Этап 2 - Синтаксический анализ (Parsing):
    Последовательность токенов преобразуется в Абстрактное синтаксическое
    дерево (AST). Парсер проверяет, что программа следует синтаксическим
    правилам языка. Результат: дерево, представляющее структуру программы.

Этап 3 - Семантический анализ и проверка типов:
    Проверяется правильность использования типов, видимости переменных, и
    семантическая корректность. На этом этапе также выполняется мономорфизация
    - превращение универсальных (generic) функций в конкретные версии для каждого
    типа параметра. Результат: AST с полной информацией о типах.

Этап 4 - Кодогенерация:
    Результат этапа 3 превращается в LLVM IR (промежуточное представление LLVM).
    После этого применяются оптимизации LLVM (в зависимости от уровня оптимизации),
    и затем IR компилируется в машинный код для целевой архитектуры (x86-64 на
    Windows). Результат: исполняемый файл.

2.3 ПРАВИЛО TOP-DOWN И ПОРЯДОК ОПРЕДЕЛЕНИЙ

Критически важный принцип в CBlerr: функция ДОЛЖНА быть определена ПЕРЕД её
использованием. Это создаёт предсказуемый порядок исполнения и позволяет
компилятору проводить более эффективную проверку типов за один проход.

Этот принцип позволяет:
• Избежать сложной двухпроходной компиляции
• Знать на момент проверки функции все её зависимости
• Обнаружить использование неопределённой функции сразу же
• Упростить реализацию компилятора

Пример правильного порядка:

    def add(a: int, b: int) -> int:    # Определена первой
        return a + b
    
    def multiply_and_add(x: int, y: int, z: int) -> int:  # Определена второй
        return multiply(add(x, y), z)   # Вызывает add, которая уже определена
    
    def multiply(a: int, b: int) -> int:  # Определена третьей
        return a * b

Исключение для взаимной рекурсии: можно использовать extern:

    extern def function_a(x: int) -> int    # Объявление, не определение
    
    def function_b(x: int) -> int:
        if x == 0:
            return 1
        return function_a(x - 1)
    
    def function_a(x: int) -> int:          # Теперь определение
        return function_b(x * 2)

2.4 ГЛАВНАЯ ФУНКЦИЯ И ТОЧКА ВХОДА

Функция main() ВСЕГДА должна быть последней функцией в файле. Это является
точкой входа программы при её запуске. Операционная система передаёт управление
этой функции, и она является первым пользовательским кодом, который выполняется.

Возвращаемое значение main() становится кодом выхода программы:
• Возвращение 0 означает успешное завершение
• Любое ненулевое значение означает ошибку

    def helper1() -> int:
        return 42
    
    def helper2() -> void:
        printf("Helper 2\n".data as *void)
    
    def main() -> int:
        helper2()
        return helper1()
    # main ВСЕГДА в конце!

        while i <= n:
            if is_prime(i):
                printf("%d ".data as *void, i)
            i = i + 1
        printf("\n".data as *void)
        return 0

Сортировка массива:
    extern def printf(fmt: *void, ...) -> int
    
    def bubble_sort(arr: *int, n: int) -> void:
        i: int = 0
        while i < n:
            j: int = 0
            while j < n - i - 1:
                if arr[j] > arr[j + 1]:
                    temp: int = arr[j]
                    arr[j] = arr[j + 1]
                    arr[j + 1] = temp
                j = j + 1
            i = i + 1
    
    def main() -> int:
        arr: *int = [64, 34, 25, 12, 22, 11, 90, 88]
        n: int = 8
        bubble_sort(arr, n)
        i: int = 0
        while i < n:
            printf("%d ".data as *void, arr[i])
            i = i + 1
        printf("\n".data as *void)
        return 0

Тип     | Размер  | Знаковый | Диапазон                           | Применение

i8      | 8 бит   | Да       | -128 до 127                        | Символы, байты
i16     | 16 бит  | Да       | -32,768 до 32,767                 | Экономия памяти

Выбор типа критичен для производительности и корректности:

    small_value: u8 = 255                      # Экономит память

    large_value: i64 = 9223372036854775807     # Максимальное 64-битное значение

    # Типичное использование в циклах
    for i = 0; i < 256; i = i + 1:             # i: int подразумевается
        process_byte(i as u8)

Переполнение при работе с целыми числами:
- Беззнаковое переполнение: 255 + 1 = 0 (для u8)
- Знаковое переполнение: 127 + 1 = -128 (для i8) - это неопределённое поведение!

4.1 ПОЛНАЯ ДЕКЛАРАЦИЯ И ИНИЦИАЛИЗАЦИЯ

Все переменные в CBlerr требуют явной типизации. Это делает код более понятным
и позволяет компилятору выполнять лучшую проверку типов:

    x: int = 10                         # Локальная переменная с инициализацией
    name: str = "Alice"                 # Строковая переменная
    active: bool = true                 # Логическая переменная
    value: f64 = 3.14                  # Число с плавающей точкой
    ptr: *int = 0                       # Указатель на int

Если инициальное значение не предоставлено, переменная содержит неопределённое
значение (остатки памяти):

    uninitialized: int                  # ОПАСНО! Содержит мусор!
    # Нужно использовать...
    initialized: int = 0                # Правильно: явно инициализировано

4.2 ИЕРАРХИЯ ОБЛАСТИ ВИДИМОСТИ

В CBlerr существует три вида области видимости:

Глобальная область видимости (весь файл):
    x: int = 0              # Глобальная переменная, видна везде

    def function_one() -> void:
        printf("%d\n".data as *void, x)  # Может использовать x

Локальная область видимости (внутри функции или блока):
    def function_two() -> void:
        local_var: int = 5              # Видна только внутри function_two
        printf("%d\n".data as *void, local_var)
        
        if true:
            nested_var: int = 10        # Видна только в этом блоке
            printf("%d\n".data as *void, nested_var)

Параметры функции:
    def add(a: int, b: int) -> int:     # a и b видны только здесь
        return a + b

Правило затенения (shadowing):
    x: int = 5                          # Глобальная переменная

    def shadow_test() -> void:
        x: int = 10                     # Локальная x затеняет глобальную
        printf("%d\n".data as *void, x) # Выведет 10, а не 5!

4.3 КОНСТАНТЫ И COMPILE-TIME ВЫЧИСЛЕНИЯ

    const PI: f64 = 3.14159265359      # Математическая константа
    const MAX_PLAYERS: int = 16         # Максимальное число игроков
    const GRAVITY: f64 = 9.81           # Ускорение свободного падения
    const SCREEN_WIDTH: int = 1920
    const SCREEN_HEIGHT: int = 1080

Константы имеют особенности:

• Обычно должны быть инициализированы литеральными значениями (для глобальных)
• Используются компилятором для оптимизации
• Часто используются в match выражениях для проверки значений

    if x < y:           # Меньше (для целых, float)
        action()

        # Правильное сравнение с epsilon

5.3 ЛОГИЧЕСКИЕ ОПЕРАТОРЫ И SHORT-CIRCUIT EVALUATION

    if (condition1) and (condition2):
        # Выполняется если оба условия истинны
        # Если condition1 ложно, condition2 не проверяется
        action()
    
    if (condition1) or (condition2):
        # Выполняется если хотя бы одно условие истинно
        # Если condition1 истинно, condition2 не проверяется
        action()
    
    if not condition:
        # Выполняется если условие ложно

5.4 ПОЛНЫЙ НАБОР ПОБИТОВЫХ ОПЕРАТОРОВ

    flags: int = 0xFF & 0x0F        # Побитовое И = 15 (0x0F)
    mask: int = 1 << 4              # Левый сдвиг = 16
    value: int = 10 >> 1            # Правый сдвиг = 5
    xor_result: int = 12 ^ 10       # XOR (исключающее ИЛИ) = 6
    or_result: int = 12 | 10        # Побитовое ИЛИ = 14
    inverted: int = ~5              # Побитовое НЕ = -6

Использование побитовых операций:
- Флаги: (flags & 0x01) != 0 - проверить бит
- Установка бита: flags | 0x01
- Очистка бита: flags & ~0x01
- Переключение бита: flags ^ 0x01

5.5 ОПЕРАТОРЫ ПРИСВАИВАНИЯ И СОСТАВНЫЕ ОПЕРАТОРЫ

    x: int = 10

    x *= 2                          # x = x * 2 = 24
    x /= 4                          # x = x / 4 = 6
    x %= 3                          # x = x % 3 = 0
    
    flags: int = 0xFF
    flags &= 0x0F                   # flags = flags & 0x0F
    flags |= 0x10                   # flags = flags | 0x10
    flags ^= 0x01                   # flags = flags ^ 0x01

            j = 0

22.1 ВСТРОЕННЫЕ ФУНКЦИИ (@inline)

Использование @inline для часто вызываемых функций может значительно улучшить
производительность, так как компилятор будет вставлять код функции вместо её
вызова.

    @inline
    def clamp(value: int, min: int, max: int) -> int:
        if value < min:
            return min
        if value > max:
            return max
        return value
    
    @inline
    def absolute(x: int) -> int:
        if x < 0:
            return -x
        return x

22.2 ПОБИТОВЫЕ ОПЕРАЦИИ

Побитовые операции работают намного быстрее, чем арифметические:

    # Быстро: побитовое И, ИЛИ, XOR
    mask: int = 0xFF & value               # Маскирование
    
    # Очень быстро: сдвиги вместо умножения/деления
    doubled: int = value << 1              # Умножить на 2
    halved: int = value >> 1               # Разделить на 2
    
    # Быстро: проверка степени двойки
    if (value & (value - 1)) == 0:
        printf("Value is power of 2\n".data as *void)

        s.top = s.top + 1

Быстрая сортировка (QuickSort):

    def partition(arr: *int, low: int, high: int) -> int:
        pivot: int = arr[high]
        i: int = low - 1
        j: int = low
        while j < high:
            if arr[j] < pivot:
                i = i + 1
                temp: int = arr[i]
                arr[i] = arr[j]
                arr[j] = temp
            j = j + 1
        temp: int = arr[i + 1]
        arr[i + 1] = arr[high]
        arr[high] = temp
        return i + 1
    
    def quicksort(arr: *int, low: int, high: int) -> void:
        if low < high:
            pi: int = partition(arr, low, high)
            quicksort(arr, low, pi - 1)
            quicksort(arr, pi + 1, high)

Слияние отсортированных массивов (Merge):

    def merge_sorted(left: *int, left_size: int, right: *int,
                     right_size: int, result: *int) -> void:
        i: int = 0
        j: int = 0
        k: int = 0
        
        while i < left_size and j < right_size:
            if left[i] <= right[j]:
                result[k] = left[i]
                i = i + 1
            else:
                result[k] = right[j]
                j = j + 1
            k = k + 1
        
        while i < left_size:
            result[k] = left[i]
            i = i + 1
            k = k + 1
        
        while j < right_size:
            result[k] = right[j]
            j = j + 1
            k = k + 1

ОБЪЕКТНО-ОРИЕНТИРОВАННЫЙ ПАТТЕРН:

    struct Vector:
        x: f64
        y: f64
        z: f64
    
    def vector_create(x: f64, y: f64, z: f64) -> Vector:
        v: Vector
        v.x = x
        v.y = y
        v.z = z
        return v
    
    def vector_magnitude(v: *Vector) -> f64:
        return sqrt(v.x * v.x + v.y * v.y + v.z * v.z)
    
    def vector_dot(v1: *Vector, v2: *Vector) -> f64:
        return v1.x * v2.x + v1.y * v2.y + v1.z * v2.z

В CBlerr каждый тип данных имеет определённое представление в памяти и определенные
операции, которые можно выполнять. Правильное использование типов критично для
написания надёжного кода.

Для целых чисел:
- Знаковые типы используют представление в дополнительном коде (two's complement)
- Целочисленные операции являются точными (в пределах диапазона типа)
- Переполнение является неопределённым поведением и может привести к непредсказуемым результатам

Для чисел с плавающей точкой:
- Используется стандарт IEEE 754
- Операции не являются полностью ассоциативными (порядок влияет на результат)
- Сравнение на точное равенство опасно из-за ошибок округления

Функциональное программирование:

Процедурное программирование:

- Возможность модификации переменных

Объектно-ориентированное программирование:

- Наследование можно эмулировать через вложенные структуры

СИСТЕМА ТИПОВ И ТИПИЗАЦИЯ:

Статическая типизация:
- Все типы определяются на этапе компиляции
- Компилятор проверяет совместимость типов
- Ошибки типов обнаруживаются до запуска программы

Явная типизация:
- Все переменные должны иметь явно указанный тип
- Преобразования типов выполняются явно (через as)
- Нет неявных преобразований, которые могут скрыть ошибки

Универсальные типы (generics):
- Функции и структуры могут быть параметризованы типом
- Компилятор автоматически создаёт специализированные версии
- Позволяет писать универсальный код без потери типизации

УПРАВЛЕНИЕ РЕСУРСАМИ И ПАМЯТЬЮ:

Выделение памяти:
- malloc() выделяет память динамически
- Результат ДОЛЖЕН быть проверен на NULL
- Размер должен быть достаточным для всех данных

Освобождение памяти:
- free() должна быть вызвана для каждого malloc()
- Двойное освобождение приводит к undefined behavior
- Использование освобождённой памяти опасно

Утечки памяти:
- Возникают когда malloc-ed память не освобождается
- Критично избегать в долгоживущих приложениях
- Каждый путь выполнения должен освобождать выделенную память

ОБРАБОТКА ОШИБОК И НАДЁЖНОСТЬ:

Стратегии обработки ошибок:
- Возврат кодов ошибок для функций
- Глобальное состояние ошибок для асинхронных ошибок
- Проверка предусловий перед выполнением операций
- Проверка постусловий после выполнения операций

Защитное программирование:
- Всегда проверяйте результаты malloc()
- Всегда проверяйте границы массивов
- Всегда валидируйте входные данные
- Используйте assert-подобные проверки в критичных местах

Логирование и отладка:
- printf() для вывода информации отладки
- Детальные сообщения об ошибках для диагностики
- Уровни детальности логирования

ПРОИЗВОДИТЕЛЬНОСТЬ И ОПТИМИЗАЦИЯ:

Профилирование перед оптимизацией:
- Измеряйте время выполнения критичных функций
- Найдите реальные узкие места
- Оптимизируйте только после профилирования

Методы оптимизации:
- Встроенные функции (@inline) для часто вызываемого кода
- Побитовые операции вместо арифметических
- Кэширование значений для избежания повторных вычислений
- Правильная организация данных для лучшей локальности кэша

Компиляция с оптимизацией:
- -O0: нет оптимизации (быстрая компиляция)
- -O1: базовая оптимизация
- -O2: средняя оптимизация (рекомендуется)
- -O3: агрессивная оптимизация (может увеличить размер кода)
