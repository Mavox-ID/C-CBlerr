# Руководство по надёжному написанию кода на языке CBlerr

**Версия:** 1.0 (Основано на текущей реализации LLVM и C-бэкендов)  
**Аудитория:** Разработчики и AI-ассистенты (LLMs).

Этот документ — строгий практический гайд по написанию кода на CBlerr. Ввиду особенностей текущей реализации лексера, AST и двойного бэкенда (LLVM JIT / C-Standalone), корректный синтаксически код может вести себя непредсказуемо, если нарушить внутренние контракты компилятора.

---

## 1. Синтаксис и форматирование

- **Отступы:** Используйте строго 4 пробела. Смешивание табов и пробелов сломает лексер.
- **Двоеточия:** Блочные конструкции (`def`, `if`, `while`, `for`, `struct`) **обязаны** завершаться двоеточием `:`.
- **Пустые массивы:** Не поддерживаются. Компилятор не сможет вывести тип.

```flux
# [ПЛОХО]
arr = [] 
if x > 0 # Ошибка: нет двоеточия
    print(x)

# [ХОРОШО]
arr: array<int> = [0]
if x > 0:
    print(x)
```

## 2. Порядок функций (Особенность LLVM)

LLVM-бэкенд компилирует функции последовательно сверху вниз. Если функция `A` вызывает функцию `B`, то `B` должна быть объявлена **выше** `A` в коде (или объявлена как `extern`).

```flux
# [ПЛОХО] - LLVM выдаст "Function 'do_math' not found"
def main() -> int:
    return do_math(5)

def do_math(x: int) -> int:
    return x + 1

# [ХОРОШО]
def do_math(x: int) -> int:
    return x + 1

def main() -> int:
    return do_math(5)
```

## 3. Управление памятью (Смертельная ловушка `alloca`)

Внутри функций литералы массивов выделяются на стеке (`alloca`). **Никогда** не возвращайте указатель на локальный массив, он будет уничтожен при выходе из функции.

```flux
# [ПЛОХО] - Возврат мусорного указателя (Segfault)
def get_data() -> *int:
    arr: array<int> = [1, 2, 3]
    return &arr

# [ХОРОШО] - Выделение в куче (Heap)
extern def malloc(size: i32) -> *void

def get_data() -> *int:
    ptr: *int = malloc(12) as *int
    # ... ручное заполнение памяти ...
    return ptr
```

## 4. Строки и взаимодействие с C / WinAPI

В текущем компиляторе тип `str` под капотом является структурой:
`typedef struct { const char* data; int64_t length; } flux_string;`

Чтобы передать строку в C-функцию или WinAPI, которая ожидает `char*`, **обязательно** обращайтесь к полю `.data` и кастуйте его.

```flux
# [ПЛОХО] - WinAPI получит структуру вместо адреса текста
extern def MessageBoxA(hWnd: *void, text: *void, cap: *void, t: int) -> int
MessageBoxA(0 as *void, "Text" as *void, "Cap" as *void, 0)

# [ХОРОШО] - Передача C-строки
extern def MessageBoxA(hWnd: *void, text: *void, cap: *void, t: int) -> int

def main() -> int:
    txt: str = "Text"
    cap: str = "Cap"
    MessageBoxA(0 as *void, txt.data as *void, cap.data as *void, 0)
    return 0
```

## 5. Булева логика (Отсутствие Short-Circuit)

Операторы `and` и `or` вычисляют **обе части выражения**. Короткого замыкания нет!

```flux
# [ПЛОХО] - Если ptr == null, код упадет при попытке разыменовать правую часть!
if (ptr != 0 as *void) and (*ptr > 0):
    do_something()

# [ХОРОШО] - Вложенные проверки
if ptr != 0 as *void:
    if *ptr > 0:
        do_something()
```

## 6. Работа с `float` (Ограничения LLVM)

В текущем LLVM-бэкенде прямое сравнение (`==`, `<`, `>`) типов `float` вызывает `NotImplemented`.

```flux
# [ПЛОХО] - Упадет при компиляции LLVM
if f1 > f2:
    print("Bigger")

# [ХОРОШО] - Обходные пути
# Вариант А: Использовать Standalone C-бэкенд (там сработает)
# Вариант Б: Привести к int (если важна только целая часть)
if (f1 as int) > (f2 as int):
    print("Bigger")
```

## 7. Структуры и указатели

Предпочитайте передавать и хранить структуры как указатели (`*MyStruct`). LLVM ожидает указательную семантику при доступе к полям (через GEP инструкции). Если вы обращаетесь к полю структуры, убедитесь, что тип переменной явно указан.

```flux
struct Point:
    x: int
    y: int

# [ХОРОШО]
def move(p: *Point):
    p.x = p.x + 1
```

## 8. Глобальные переменные

Инициализация глобальных переменных поддерживает **только литералы** (числа и строки). Сложную логику переносите в функцию `main`.

```flux
# [ПЛОХО]
const MY_PTR: *void = malloc(100) # Ошибка компиляции

# [ХОРОШО]
const MAX_SPEED: int = 100
global_ptr: *void

def main() -> int:
    global_ptr = malloc(100)
    return 0
```

9. Эталонный пример кода на CBlerr
Этот пример демонстрирует все лучшие практики языка: объявление extern функций, работу со структурами, выделение памяти в куче (malloc), правильную передачу строк в C-функции (через .data as *void) и отсутствие and для защиты от Null-указателей.
code
Flux
# 1. Сначала объявляем все зависимости из C (libc)
extern def fopen(filename: *void, mode: *void) -> *void
extern def fclose(stream: *void) -> int
extern def malloc(size: i32) -> *void
extern def free(ptr: *void) -> void
extern def puts(s: *void) -> int

# 2. Объявляем структуры
struct FileContext:
    handle: *void
    is_open: bool

# 3. Вспомогательные функции (выше тех, кто их использует)
def create_context() -> *FileContext:
    # Используем выделение в куче (heap), а не локальные переменные!
    # Предположим, размер структуры 16 байт
    ptr: *FileContext = malloc(16) as *FileContext
    ptr.is_open = false
    ptr.handle = 0 as *void
    return ptr

def open_log(filename: str) -> *FileContext:
    ctx: *FileContext = create_context()
    mode: str = "w"
    
    # ПРАВИЛЬНО: Передаем C-функциям поле .data от строки
    ctx.handle = fopen(filename.data as *void, mode.data as *void)
    
    # ПРАВИЛЬНО: Избегаем 'and', используем вложенные if для безопасности
    if ctx.handle != 0 as *void:
        ctx.is_open = true
        
    return ctx

# 4. Главная функция всегда в самом низу
def main() -> int:
    file_name: str = "app_log.txt"
    ctx: *FileContext = open_log(file_name)
    
    if ctx.is_open:
        msg: str = "Log file created successfully!"
        # Передаем строку в puts
        puts(msg.data as *void)
        fclose(ctx.handle)
        
    # Не забываем чистить память
    free(ctx as *void)
    return 0

# 🤖 Инструкция для ИИ (System Prompt Rules)

**Внимание контексту LLM:** Если ты генерируешь код на языке CBlerr, ты **обязан** строго следовать правилам ниже. Это не Python и не C++, это кастомный язык со своими AST-контрактами.

### Твои строгие ограничения (Constraints):
1. **Явная типизация:** Все параметры функций и возвращаемые значения должны иметь типы. Не оставляй `None` в типах аргументов.
2. **Отступы:** Только 4 пробела. Никаких табов.
3. **C-Строки:** Если вызываешь `extern` C-функцию, принимающую `char*`, передавай строку как `my_string.data as *void`.
4. **Порядок функций:** Сначала определяй зависимости (или пиши `extern`), затем функции, которые их используют. `main()` всегда внизу.
5. **Short-Circuit:** Не используй `and` для защиты от null-pointer dereference. Пиши вложенные `if`.
6. **Возврат из функции:** Функции с `-> int` обязаны завершаться `return <expr>`.

### Схема AST для генерации:
Если промпт просит структуру AST, используй этот формат:
- `FunctionDef`: `{ name: str, params: [(str, Type)], return_type: Type, body: [Stmt], is_extern: bool }`
- `Assign`: `{ target: str|Expr, value: Expr, var_type: Type|null }`
- `Call`: `{ func_name: str, args: [Expr] }`

### Алгоритм самопроверки ИИ перед выдачей кода:
- [ ] Все ли функции имеют `:` после скобок?
- [ ] Определены ли вызываемые `def` выше по тексту?
- [ ] Нет ли возврата локального массива (`&arr`) из функции?
- [ ] Передается ли `.data as *void` при экспорте строк в C?

*Если код нарушает хотя бы одно правило из чеклиста — перепиши его до выдачи пользователю.*
